## The Problem UART Solves

Microcontrollers need to talk to the outside world. They need to send debug messages to your laptop, receive commands from a host, or exchange data with another chip. But how?

The simplest possible answer is two wires: one for sending (TX), one for receiving (RX). That's the heart of UART.

## What UART Actually Is

**UART** stands for *Universal Asynchronous Receiver-Transmitter*. It's not a protocol owned by a standards body — it's a hardware specification for a serial communication method that has been around since the 1960s.

Key characteristics:

- **Asynchronous** — no shared clock signal between devices
- **Full-duplex** — can send and receive simultaneously
- **Point-to-point** — connects exactly two devices

## The Hardware Interface

A UART connection uses two signal lines:

| Pin | Direction | Description |
|-----|-----------|-------------|
| TX  | Output    | Transmit data |
| RX  | Input     | Receive data |
| GND | —         | Common ground (always needed) |

> **Critical:** TX on one device connects to RX on the other, and vice versa. Connecting TX→TX is a common beginner mistake.

## How Data Moves

UART sends data one bit at a time, framed within a *UART frame*:

```
IDLE  START  D0  D1  D2  D3  D4  D5  D6  D7  STOP
HIGH    0    bit bit bit bit bit bit bit bit    1
```

- **Idle state** — the line sits HIGH when nothing is being sent
- **Start bit** — always 0, signals the receiver to start sampling
- **Data bits** — typically 8 bits, LSB first
- **Stop bit** — always 1, marks end of frame

## Baud Rate: The Timing Agreement

Since UART is asynchronous, both sides must agree on one thing: **how fast** to send bits. This is the *baud rate* — bits per second.

```c
// Common baud rates
9600    // Classic default, good for long cables
115200  // Modern default for USB-serial adapters
1000000 // 1 Mbaud — used in high-speed embedded systems
```

If the baud rates don't match, you'll receive garbage. This is the #1 cause of UART debugging headaches.

## Real Hardware Example

On an STM32 microcontroller, UART is called *USART* (the extra S stands for *Synchronous* — it can optionally use a clock). Here's the simplest possible transmit:

```c
#include "stm32f4xx_hal.h"

UART_HandleTypeDef huart2;

void uart_send_string(const char *str) {
    HAL_UART_Transmit(&huart2, (uint8_t *)str, strlen(str), HAL_MAX_DELAY);
}

int main(void) {
    HAL_Init();
    SystemClock_Config();
    MX_USART2_UART_Init();  // sets baud rate, data bits, stop bits

    while (1) {
        uart_send_string("Hello from STM32!\r\n");
        HAL_Delay(1000);
    }
}
```

## When to Use UART

UART is the right choice when you need:

- A simple debug console (connect to a USB-serial adapter)
- Communication with GPS modules, GSM modems, Bluetooth modules
- Talking to another microcontroller over a short cable

It's *not* the right choice for:
- Multiple devices on the same bus (use SPI or I2C instead)
- Very high speeds over long distances
- Synchronous data that needs a clock reference
